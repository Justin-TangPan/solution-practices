# 技术评估方法论 (8 维度评审框架)

## 概述

本文档定义华为云解决方案实践的技术评估标准框架，用于对基于 RFS + Terraform 的一键部署方案进行结构化评审。

**评估对象**：华为云 Solution as Code 方案（RFS 模板 + user_data 脚本 + 部署文档）
**评估目的**：识别技术风险、量化方案质量、输出优化建议
**适用范围**：方案开发阶段评审 / 发布前终审 / 存量方案巡检

---

## 1. 8 维度评估框架

### 1.1 评审维度总览

```
┌─────────────────────────────────────────────────────────────────┐
│                    8 维度技术评估框架                            │
├───────────┬───────────┬───────────┬───────────┬───────────┬────┤
│ 技术可行性  │ 部署效率   │ 运维便捷性  │ 脚本兼容性  │ TF通用性   │
│ ★★★★★     │ ★★★★★     │ ★★★★☆     │ ★★★★☆     │ ★★★★★     │
├───────────┼───────────┼───────────┼───────────┼───────────┤
│ RFS稳定性  │ 集群能力   │ 回滚机制   │           │           │
│ ★★★★★     │ ★★★☆☆     │ ★★★★☆     │           │           │
└───────────┴───────────┴───────────┴───────────┴───────────┘
```

### 1.2 各维度详解

#### 维度 1: 技术可行性

**权重**：★★★★★（一票否决）

**评估目标**：确认方案使用的所有云服务均已商用、无技术依赖黑洞

| 检查项 | 通过标准 | 否决条件 |
|--------|---------|---------|
| 云服务状态 | 全部已商用 | 使用邀测/公测服务 |
| Provider 支持 | huaweicloud provider 可管理 | 资源不存在对应 TF resource |
| 区域可用性 | 目标区域有全部服务 | 某服务在目标区域不可用 |
| ECS 规格 | 目标规格在区域有库存 | GPU 规格在区域无库存 |
| 镜像兼容 | 脚本在目标 OS 已验证 | 未验证或已知不兼容 |
| API 版本 | 使用最新稳定 API 版本 | 使用已弃用 API |
| 依赖开源 | 开源组件可正常获取 | 依赖被墙/下架的源 |

**评分标准**：
| 分数 | 条件 |
|------|------|
| ★★★★★ | 全部通过，无已知风险 |
| ★★★★☆ | 全部通过，有低风险项（如部分区域不可用） |
| ★★★☆☆ | 存在 1 项否决条件，但有替代方案 |
| ★★☆☆☆ | 存在 2+ 项否决条件 |
| ★☆☆☆☆ | 核心依赖不可用 |

#### 维度 2: 部署效率

**权重**：★★★★★

**评估目标**：评估一键部署全链路耗时和自动化程度

| 检查项 | 通过标准 | 扣分项 |
|--------|---------|--------|
| 部署总耗时 | < 15 分钟 | 超过 20 分钟 |
| user_data 执行 | 自动完成无需人工介入 | 需要手动干预 |
| 模型/数据下载 | 使用镜像源或 OBS 加速 | 直接依赖 GitHub/外网 |
| 后台任务 | 不阻塞主部署流程 | 串行等待导致耗时翻倍 |
| 幂等部署 | 多次执行结果一致 | 非幂等导致失败 |
| 进度可视化 | RFS 事件可跟踪 | user_data 黑盒不可见 |

**瓶颈分析方法**：
```
总耗时 = max(资源创建时间, user_data 执行时间)

资源创建瓶颈:
  └─ ECS 创建: 30-90s (区域库存影响)
  └─ RDS 创建: 5-15min (主备模式更慢)
  └─ GPU ECS: 可能需审批

user_data 瓶颈:
  └─ apt/yum update: 60-180s → 华为云镜像加速可优化
  └─ 模型下载 (Ollama pull): 2-5min → OBS 分发可优化
  └─ Docker pull: 60-120s → SWR 镜像加速可优化
  └─ 源码编译: 5-20min → 预编译 Release 可优化
```

#### 维度 3: 运维便捷性

**权重**：★★★★☆

**评估目标**：评估方案上线后的运营维护体验

| 检查项 | 权重 | 最佳实践 |
|--------|------|---------|
| 服务托管 | 高 | systemd 托管，Restart=always |
| 日志输出 | 高 | 统一日志目录 + 轮转 |
| 监控告警 | 中 | 对接 CES + SMN |
| 备份恢复 | 中 | CBR 策略自动备份 |
| 扩缩容 | 低 | AS 弹性伸缩（>= 生产级） |
| 版本升级 | 低 | 脚本化升级流程 |
| 成本管理 | 中 | 资源 tags 标记 + Cost Center |

#### 维度 4: 脚本兼容性

**权重**：★★★★☆

**评估目标**：评估 user_data 脚本在不同 OS 和环境的可移植性

**OS 兼容矩阵**：

| OS | 包管理器 | Docker | Nginx | systemd | Python | Ollama | cloud-init |
|---|---------|--------|-------|---------|--------|--------|------------|
| Ubuntu 22.04 | apt | apt install | apt install | ✅ | 3.10 | ✅ | ✅ |
| Ubuntu 20.04 | apt | apt install | apt install | ✅ | 3.8 | ✅ | ✅ |
| CentOS 7.9 | yum | repo+install | yum install | ✅ | 3.6 | ❌ | ✅ |
| CentOS 8.4 | dnf | repo+install | dnf install | ✅ | 3.8 | ❌ | ✅ |
| Anolis OS 8.8 | yum | repo+install | yum install | ✅ | 3.8 | ✅ | ✅ |
| openEuler 22.03 | yum | repo+install | yum install | ✅ | 3.9 | ✅ | ✅ |

**评分标准**：

| 分数 | 条件 |
|------|------|
| ★★★★★ | 声明兼容 4+ OS，全部通过测试 |
| ★★★★☆ | 声明兼容 2-3 OS，有测试结果 |
| ★★★☆☆ | 声明兼容 1 OS，无测试 |
| ★★☆☆☆ | 仅支持特定版本，无分支处理 |
| ★☆☆☆☆ | 脚本有硬编码路径/版本，不可移植 |

#### 维度 5: TF 模板通用性

**权重**：★★★★★

**评估目标**：评估 Terraform 模板的解耦度、参数化水平和可复用性

| 检查项 | 通过标准 |
|--------|---------|
| 参数化 | 所有可变值通过 variable 定义 |
| 敏感信息 | 密码/密钥标记 `sensitive = true` |
| 模块化 | 复杂方案拆分为 modules |
| 条件支持 | 使用 count/for_each/condition |
| 资源标记 | 所有资源有 tags 管理 |
| 输出完整 | 用户需要的信息都在 outputs |
| 依赖隐式 | 不使用 `depends_on` |
| Provider | source/version 正确声明 |

**模板通用性分级**：

```
Level 1: 单文件 main.tf + variables.tf          → 基础可用
Level 2: 拆分 outputs.tf + providers.tf          → 标准结构
Level 3: + .extension 分组                       → 生产级 UI
Level 4: + modules/ 模块化                       → 企业级可复用
Level 5: + 条件表达式 + for_each + terratest     → 极致工程化
```

#### 维度 6: RFS 编排稳定性

**权重**：★★★★★

**评估目标**：评估资源编排链的健壮性和容错能力

**风险分析框架**：

| 风险场景 | 概率 | 影响 | 应对 |
|---------|------|------|------|
| 区域资源不足 | 低 | 高 | 多可用区 fallback |
| API 限流 | 低 | 中 | RFS 内置重试 |
| Provider 版本兼容 | 低 | 高 | 固定 version 约束 |
| 资源创建超时 | 中 | 中 | 增加超时配置 |
| 依赖循环 | 低 | 高 | 避免循环引用 |
| 并行创建冲突 | 低 | 中 | 隐式依赖解决 |

#### 维度 7: 批量集群能力

**权重**：★★★☆☆

**评估目标**：评估方案是否支持水平扩展和大规模部署

| 规模 | 推荐方式 | TF 管理 | 部署时间 |
|------|---------|---------|---------|
| 1-5 节点 | count / for_each | 直接管理 | 10-25min |
| 5-20 节点 | count + 并行子网 | 直接管理 | 20-60min |
| 20-100 节点 | AS 弹性伸缩组 | 仅管理 AS 配置 | 按需弹性 |
| 100+ 节点 | AS + 多可用区 | 仅管理模板 | 按需弹性 |

#### 维度 8: 故障回滚机制

**权重**：★★★★☆

**评估目标**：评估方案在部署失败时的数据保护和恢复能力

| 故障层级 | RFS 回滚 | 数据保护 | 说明 |
|---------|---------|---------|------|
| 网络资源创建失败 | 自动回滚 | 无损 | RFS 原生能力 |
| ECS 创建失败 | 自动回滚 | 无损 | 关联资源被回收 |
| user_data 失败 | ❌ 无法感知 | ECS 已创建 | **最大风险点** |
| 应用启动失败 | ❌ 无法感知 | 不影响云资源 | systemd 自愈 |
| 运行时故障 | 不涉及 | 数据可能受损 | 需 CBR 备份 |

**user_data 失败应对等级**：

| 等级 | 措施 | 复杂度 |
|------|------|--------|
| L0 | 无处理（当前默认） | 无 |
| L1 | 脚本内 `set -e` + 日志 | 低 |
| L2 | checkpoint 断点续执行 | 中 |
| L3 | health check + SMN 告警 | 中 |
| L4 | 自动回滚 + 清理已创建资源 | 高 |

## 2. 评估流程标准

### 2.1 流程时序

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│  需求   │──▶│  逐维度  │──▶│  瓶颈   │──▶│  评分   │──▶│  优缺   │──▶│  报告   │
│  解析   │   │  分析    │   │  识别    │   │  汇总   │   │  建议   │   │  输出   │
└─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘
```

### 2.2 输入与产出

**输入**：
- Terraform 模板文件集（.tf / .tf.json）
- user_data 脚本（.sh）
- 部署文档（README.md）
- 方案描述 / 架构图
- 目标部署环境信息（区域、规格）

**产出**：
- 完整评审报告（9 章节）
- 8 维度评分雷达图（ASCII）
- 瓶颈列表 + 优化建议
- P0-P3 改造优先级

## 3. 报告结构标准

评审报告必须包含以下 9 个章节：

| # | 章节 | 核心内容 |
|---|------|---------|
| 1 | 核心架构总览 | 分层架构图、数据流、组件职责 |
| 2 | 流程架构梳理 | 部署流程图、RFS 内部时序、耗时分布 |
| 3 | TF 模板编写要点 | 格式选型、文件组织、resource 规范 |
| 4 | RFS 调用配置参数 | 参数配置表、.extension 分组、委托 |
| 5 | user_data 脚本注入规范 | 幂等模式、错误处理、兼容矩阵 |
| 6 | 一键部署执行逻辑 | 状态机、事件序列、超时重试 |
| 7 | 多维度技术评审 | 8 维度评分 + 详细分析 + 瓶颈 |
| 8 | 落地注意事项 | 网络安全、费用、区域选择 |
| 9 | 优化改造方案 | P0-P3 分级优化 + 目标架构 |

## 4. 逐项检查清单

### 4.1 TF 模板检查

- [ ] `versions.tf` — provider source/version 正确声明
- [ ] `providers.tf` — 无硬编码 AK/SK
- [ ] `variables.tf` — 每个 variable 有 description + type
- [ ] `variables.tf` — 敏感字段设置 sensitive = true
- [ ] `variables.tf` — metadata.group 与 .extension 对应
- [ ] `main.tf` — 使用隐式依赖
- [ ] `main.tf` — 多实例用 count/for_each
- [ ] `main.tf` — 所有资源有 tags 标记
- [ ] `outputs.tf` — 覆盖用户需要的信息
- [ ] `.extension` — grouping key 与 metadata.group 一致

### 4.2 脚本检查

- [ ] 以 `#!/bin/bash` 开头，`set -euo pipefail`
- [ ] 日志重定向：`exec > >(tee -a "$LOG_FILE") 2>&1`
- [ ] 变量外部化：`PARAM="${PARAM:-default}"`
- [ ] 幂等标记：检测安装标记，跳过重复执行
- [ ] OS 分支：`case "$OS_FAMILY" in ... esac`
- [ ] 错误处理：致命错误 exit，非致命 warn 继续
- [ ] 超时控制：长时间操作设置 timeout
- [ ] 后台任务：不阻塞部署流程
- [ ] 输出摘要：打印关键信息便于用户确认

### 4.3 文档检查

- [ ] README 包含方案概述
- [ ] 包含架构图（或架构描述）
- [ ] 包含资源清单表
- [ ] 包含部署步骤（前置条件 + 参数说明）
- [ ] 包含开始使用指南
- [ ] 包含预估费用
- [ ] 包含快速卸载步骤

### 4.4 安全检查

- [ ] 无硬编码 AK/SK/密码
- [ ] 安全组规则遵循最小权限
- [ ] 数据库端口不对公网开放
- [ ] admin 密码满足复杂度要求
- [ ] 不使用默认/弱口令
