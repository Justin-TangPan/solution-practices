# 解决方案实践业务评估

> **定位**：在技术评估之前，先回答"值不值得做"。
>
> **核心原则**：技术可行性 ≠ 商业可行性。一个项目技术上能部署到云上，不代表它有云上价值。

---

## 评估模型：四维雷达

每个维度 0-10 分，加权求和得出总评分。

| 维度 | 权重 | 核心问题 | 致命否决条件 |
|------|------|----------|-------------|
| **D1: 服务端属性** | 25% | 项目本质是服务端软件还是客户端工具？ | 纯桌面/GUI 工具 → 直接否决 |
| **D2: 营销价值** | 25% | 能否为华为云市场带来差异化、获客、品牌增益？ | 华为云市场已有同类且更强 → 低分 |
| **D3: 场景价值** | 25% | 是否解决真实客户痛点？客户有无付费意愿？ | 找不到愿意付费的客户场景 → 低分 |
| **D4: 云上部署价值** | 25% | 部署到华为云后是否产生增量价值（vs 本地部署）？ | 云上部署与本地部署无差异 → 0分 |

### 评分等级

| 总分 | 等级 | 建议 |
|------|------|------|
| 8-10 | 🟢 **强烈推荐** | 立即启动解决方案实践开发 |
| 6-7.9 | 🟡 **值得做** | 启动开发，但需注意短板维度的风险缓解 |
| 4-5.9 | 🟠 **勉强可做** | 仅在有战略卡位需求时启动，需重新定位 |
| 0-3.9 | 🔴 **不建议** | 放弃或重新定义方案范围 |

## D1: 服务端属性评估

### 判定标准

| 类型 | 特征 | 得分 | 云上可行性 |
|------|------|------|-----------|
| **纯服务端** | REST API / gRPC / Web 服务 / 数据库 / 消息队列 | 10 | ✅ 天然适合 |
| **服务端 + Web UI** | 后端 API + 浏览器管理面板 | 9 | ✅ 适合 |
| **CLI / SDK 工具** | 命令行工具、开发库、编译器 | 6 | ⚠️ 需包装为服务才有云价值 |
| **CLI 包装 REST API** | 在已有 API 外套一层 CLI | 3 | ❌ 云上冗余 |
| **桌面 GUI 工具** | 需要 X11/Wayland/显示器 | 0 | ❌ 无法上云 |

### 评估检查项

- [ ] 项目是否提供 REST API / gRPC / WebSocket 服务？
- [ ] 项目是否可以 headless（无显示器）运行？
- [ ] 项目是否已有 Docker / systemd 部署方式？
- [ ] 项目是否是 CLI 工具？如果是，它包装的是否是已有 API？
- [ ] 项目是否需要 GPU？如果是，华为云是否有对应 GPU 实例？

### 典型项目判定

| 项目 | 类型 | D1 得分 | 理由 |
|------|------|:-------:|------|
| Dify | 服务端 + Web UI | 9 | REST API + 管理面板，Docker 部署 |
| Ollama | 服务端 + CLI | 9 | REST API（11434），headless 运行 |
| n8n | 服务端 + Web UI | 9 | REST API + 工作流编辑器 |
| LiteLLM | 服务端 | 10 | 纯 API 代理网关 |
| ComfyUI | 服务端 + Web UI | 8 | REST API + 图形编辑器，需 GPU |
| CLI-Anything | CLI 包装层 | 3 | 在已有 API 外套 Click CLI，云上冗余 |
| GIMP | 桌面 GUI | 0 | 需要 X11，无法上云 |
| Blender | 桌面 GUI | 0 | 需要 OpenGL + 显示器 |

## D2: 营销价值评估

### 判定标准

| 等级 | 特征 | 得分 |
|------|------|------|
| **强差异化** | 华为云市场零同类，且项目本身高星标/高知名度 | 10 |
| **有差异化** | 华为云市场无同类，但项目知名度一般 | 7-8 |
| **同质但有优势** | 华为云市场有同类，但本项目有明显优势（星标/功能/性能） | 5-6 |
| **同质化** | 华为云市场已有同类且更强 | 3-4 |
| **无营销价值** | 不面向企业客户，或属于内部工具 | 1-2 |

### 评估检查项

- [ ] 华为云市场/解决方案实践库中是否有同类方案？
- [ ] 项目 GitHub Stars 数量？（>10k 为强信号）
- [ ] 项目是否被主流技术媒体/社区报道过？
- [ ] 项目是否在 Gartner/Forrester 等分析机构视野中？
- [ ] 方案定位是否能吸引新客户群体（非华为云现有客户）？
- [ ] 方案是否能与华为云自有服务（ModelArts、CCE、RDS 等）形成组合销售？

### 营销价值加分项

| 加分项 | 加分 |
|--------|------|
| 10k+ GitHub Stars | +1 |
| 50k+ GitHub Stars | +2 |
| 华为云市场零同类 | +2 |
| 可与 ModelArts/CCE/RDS 组合销售 | +1 |
| 面向 AI/ML 热门赛道 | +1 |

## D3: 场景价值评估

### 判定标准

| 等级 | 特征 | 得分 |
|------|------|------|
| **刚需场景** | 解决明确的客户痛点，客户有强付费意愿 | 10 |
| **有价值场景** | 解决真实问题，但付费意愿中等 | 7-8 |
| **锦上添花** | 有用但非必需，客户更倾向免费方案 | 5-6 |
| **小众场景** | 仅少数客户需要，市场规模有限 | 3-4 |
| **伪需求** | 看起来有用但实际无客户愿意为此付费 | 1-2 |

### 评估检查项

- [ ] 能否描述 3 个具体的客户使用场景（含客户类型、痛点、方案如何解决）？
- [ ] 客户当前如何解决这个痛点？（替代方案是什么？）
- [ ] 本方案相比替代方案的核心优势是什么？
- [ ] 客户愿意为这个方案付费吗？预估月费多少？
- [ ] 这个场景的市场规模（TAM）是否足够大？

### 场景真实性检验

**伪需求信号**（出现任一条即扣分）：

| 伪需求信号 | 扣分 |
|-----------|------|
| "让 AI 能操作 [具体系统]" — 但客户当前不需要 AI 操作该系统 | -3 |
| "统一接口" — 但客户只用 1-2 个 Provider，不需要统一 | -2 |
| "Agent-Native" — 但云服务天生 API-Native，不需要额外转换 | -3 |
| "结构化输出" — 但 `curl | jq` 已经够用 | -2 |
| "自动化" — 但客户更倾向手动控制 | -1 |

## D4: 云上部署价值评估

> **这是最关键的维度。** 很多项目技术可行但云上无增量价值。

### 判定标准

| 等级 | 特征 | 得分 |
|------|------|------|
| **云原生刚需** | 必须在云端运行（弹性伸缩、高可用、数据持久化） | 10 |
| **云上有明显增益** | 云上部署比本地有明显优势（成本/性能/可靠性） | 7-8 |
| **云上有轻微增益** | 云上部署与本地部署差异不大，但云上略方便 | 4-6 |
| **云上无差异** | 云上部署与本地部署功能完全相同 | 2-3 |
| **云上减分** | 云上部署反而不如本地（增加复杂度/延迟/成本） | 0-1 |

### 核心检验：云上增量价值清单

逐一回答以下问题，每个"是"加 1 分（基准 0 分）：

- [ ] **弹性伸缩**：云上能否按需扩缩容？本地做不到或成本极高？
- [ ] **高可用**：云上能否实现多 AZ 容灾？本地单点？
- [ ] **数据持久化**：云上 RDS/OBS 比本地存储更可靠？
- [ ] **网络效应**：云上部署后能与其他云服务（VPC/ELB/CCE）联动？
- [ ] **运维简化**：云上 RDS/CCE 托管比自建 PG/K8s 更省心？
- [ ] **成本优势**：云上按需计费比本地买服务器更便宜（对客户而言）？
- [ ] **安全合规**：云上安全组/WAF/等保比本地自建更合规？
- [ ] **全球部署**：云上多 Region 部署比本地跨地域更简单？

### 云上减分项

| 减分项 | 扣分 |
|--------|------|
| 云上只是把本地能跑的东西搬到 ECS，无任何架构变化 | -3 |
| 项目核心价值依赖桌面/GUI，云上丢失核心能力 | -5 |
| 云上部署增加额外依赖层（CLI 包装 API）但无新能力 | -3 |
| 云上延迟比本地高（如本地推理 vs 云上推理） | -1 |

### 典型项目判定

| 项目 | D4 得分 | 理由 |
|------|:-------:|------|
| Dify | 8 | 云上高可用、RDS 持久化、多 Region、与 ModelArts 联动 |
| Ollama | 6 | 云上 GPU 推理可行，但本地 GPU 体验更好（零延迟） |
| n8n | 8 | 云上 7×24 运行、RDS 持久化、与企业 VPC 联动 |
| LiteLLM | 9 | 云上是天然部署位（API 网关需 7×24 在线） |
| ComfyUI | 5 | 云上 GPU 可用但成本高，本地 GPU 更经济 |
| CLI-Anything | 1 | 云上 REST API harness 是冗余包装；桌面 harness 上不了云 |

## 与其他 Skill 的协同

| 协同场景 | 说明 |
|----------|------|
| **sac-full-pipeline 前置** | 架构师 Agent 在技术评估前先调用本 skill，若评分 < 4 则直接否决，不进入开发流程 |
| **sac-architect 决策输入** | 本 skill 的评估结果作为架构师决策的重要输入（是否启动、方案范围调整） |
| **sac-page-enhance 商品化** | 评估中的"场景价值"和"营销价值"分析可直接用于方案页面文案 |
| **sac-rfs-practices 开发** | 若 D4 评分低但决定仍做，开发时需重点设计云上增量价值（如加 HA、加 RDS、加监控） |

### 在工作流中的位置

```
用户提出方案意向
        │
        ▼
  ┌─────────────────────┐
  │ sac-business-evaluator │  ← 本 skill（商业预筛）
  │ 四维评分 → 是否值得做？ │
  └──────────┬──────────┘
             │
     评分 ≥ 6？│
     ├─ 否 → 输出否决理由，结束
     └─ 是 → 进入 sac-full-pipeline
              │
              ▼
         架构师 Agent（技术评估）
              │
              ▼
         开发 → 测试 → 安全 → 文档 → 交付
```
